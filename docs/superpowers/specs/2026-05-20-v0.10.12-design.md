**Date:** 2026-05-20
**Status:** draft

## Problem Statement

Two pain points in daily OSB operation:

1. **Manual per-runtime update on every release.** Operator runs 5-7 steps per machine to update OSB installation across all runtimes (Claude Code, Codex, Hermes, Cursor, etc.). Error-prone and tedious.
2. **No trend visibility.** Daily discipline report gives a single-day snapshot. No understanding of "how the agent learns over a week" — confidence trends, agent engagement quality, rule adoption patterns.

## Feature 1: `o2b update`

### What it does

One command that updates OSB installation across all detected runtimes by re-running the InstallAdapter pipeline with the current version's payload. Skips runtimes where the payload hasn't changed.

### User-facing surface

```bash
# Update all detected runtimes
o2b update

# Update a specific runtime
o2b update --target claudecode

# Dry-run: show what would change without applying
o2b update --dry-run

# Force apply even if payload appears unchanged
o2b update --force

# JSON output for scripting
o2b update --json
```

### Behavior

For each target (or `--target X`):

1. **Detect** — run `adapter.detect(env)` to check runtime status
2. **Plan** — generate current `McpPayload` and call `adapter.plan(payload, env)`
3. **Compare** — check sidecar manifest (`install.lock.json`) for previous payload; if identical, skip with "up-to-date" note
4. **Apply** — if payload differs (or `--force`), call `adapter.apply(plan, payload, env, opts)`
5. **Post-notes** — emit per-runtime restart hints (e.g., "restart Claude Code", `hermes gateway restart`)
6. **Verify** — run `o2b install --check` at the end for all targets

If a runtime fails to update: clear error message with hint pointing to the required manual step. No auto-recovery for cases requiring user decision.

### Exit codes

| Code | Meaning |
|------|---------|
| 0 | All targets updated or already up-to-date |
| 1 | Runtime error (I/O, subprocess failure) |
| 2 | Usage error (unknown target, bad flags) |
| 3 | `--check` found drift after update |
| 4 | User-modified-block conflict (use `--force` to override) |

### File changes

- **New:** `src/cli/update.ts` — CLI verb handler, arg parsing, orchestration loop
- **New:** `src/core/install/update.ts` — core update logic (detect → plan → compare → apply → verify)
- **New:** `tests/cli/update.test.ts`
- **New:** `tests/core/install/update.test.ts`
- **Modify:** `src/cli/main.ts` — register `update` subcommand
- **Modify:** `CHANGELOG.md` — v0.10.12 entry

### Design decisions

- **Reuse InstallAdapter machinery entirely.** No new adapter interface, no new registry. The update logic is orchestration over existing `detect/plan/apply/verify`.
- **Payload comparison via sidecar manifest.** The `install.lock.json` already stores `applied_at`, `operation`, `owned_keys`, `owned_paths`. We extend it with an additive `payload_hash` field (sha256 of the canonical McpPayload JSON) so update can skip unchanged targets without re-applying. This is backward-compatible: entries without `payload_hash` are treated as "unknown — apply on first update run", and the hash is written after the first successful update.
- **Sequential, not parallel.** Runtimes share filesystem state (the vault, the skill symlinks). Sequential execution avoids race conditions. The total time is still sub-second for most setups.
- **`--force` bypasses payload hash check.** Useful when operator knows the payload changed but the hash didn't (e.g., env variable update that doesn't affect the McpPayload struct but affects runtime behavior).

## Feature 2: Weekly Brain Digest

### What it does

Extends `brain_digest` to accept a `--window` parameter for arbitrary time windows (e.g., `7d`, `14d`, `30d`). Adds a weekly cron job that delivers a 7-day digest every Monday morning.

### CLI surface

```bash
# Default: 24h (unchanged)
o2b brain digest

# Weekly window
o2b brain digest --window 7d

# Custom window
o2b brain digest --window 14d

# JSON output
o2b brain digest --window 7d --json
```

### Window parser

Accepts `Nd` format where N is a positive integer (e.g., `1d`, `7d`, `14d`, `30d`). Bare integers without `d` suffix are also accepted (e.g., `7` → 7 days). Invalid format or non-positive values → exit 2 with error message.

### Digest content (weekly mode)

Same sections as daily, but over the 7-day window:

1. **New confirmed rules** — preferences with `confirmed_at` inside the window
2. **Confidence downshifts** — rules whose confidence_band dropped during the window
3. **Top-5 most-applied** — by evidence-application count within the window
4. **Agent quality summary** — per-agent breakdown:
   - Total brain-events (feedback + apply-evidence + note)
   - Breakdown by event type
   - How many events led to confirmed rules in the same window
   - How many events led to retired rules in the same window

### Agent quality summary implementation

Read `Brain/log/<date>.md` (or `.jsonl`) for all dates in the window. Group events by `agent` field. For each agent:

- Count events by type (`feedback`, `apply-evidence`, `note`)
- Cross-reference with `Brain/preferences/` — count preferences whose `confirmed_at` falls in the window AND whose first `applied` event matches this agent (via `evidenced_by` wikilink or log entry `preference` field)
- Cross-reference with `Brain/retired/` — count retirements in window traceable to this agent's signals

### Cron integration

**Parameterize `bin/o2b-discipline-report`:**

Add `--window` flag to the script. When set, it calls `o2b brain digest --window <value>` instead of the daily discipline report.

**New install verb:**

```bash
o2b discipline install --weekly --telegram-target telegram:-100<chat>:<topic> --at "59 8 * * 1"
```

Creates a second Hermes cron job (separate from daily) with:
- `name: "osb-weekly-brain-digest"`
- `script: "bin/o2b-discipline-report --window 7d"`
- `schedule: "59 8 * * 1"` (Monday 08:59 UTC — Monday morning in Europe/Belgrade)

### File changes

- **Modify:** `src/core/brain/digest.ts` — accept `windowDays` parameter in `RenderDigestOptions`; adjust `since` calculation; add agent quality summary section to Markdown/JSON output
- **Modify:** `src/cli/brain/verbs/digest.ts` — parse `--window` flag
- **Modify:** `bin/o2b-discipline-report` — accept `--window` flag, route to `o2b brain digest --window`
- **Modify:** `src/cli/discipline-install.ts` — add `--weekly` flag to install verb
- **New:** `src/core/brain/digest-agent-summary.ts` — agent quality summary computation (pure read over Brain/log/)
- **New:** `tests/core/brain/digest-window.test.ts`
- **New:** `tests/core/brain/digest-agent-summary.test.ts`
- **Modify:** `CHANGELOG.md`

### Design decisions

- **`--window` not `--weekly`.** More general — operator can request any window. `--weekly` would lock us into a boolean.
- **Agent quality in the digest core, not discipline.** Discipline report is about "is the agent writing to Brain at all". Digest is about "what changed in Brain". Agent quality summary belongs in digest because it references confirmed/retired rules, not just event counts.
- **Graceful degradation.** If log files are missing for some days in the window, the digest still renders with available data. No crash on partial data.

## Feature 3: Cursor SQLite Deep Parsing

### What it does

Replace the mtime-only probe for Cursor transcripts with actual SQLite parsing. Returns the number of unique sessions and total message count for the day.

### Data extracted

From `state.vscdb`:
- **Session count** — number of unique `sessionId` values with activity in the day window
- **Message count** — total number of messages/turns across all sessions in the window

No per-turn timestamps stored or returned. Just aggregate counts.

### SQLite schema assumption

Cursor uses the standard VSCode-derived `state.vscdb` with a `ItemTable` containing `key` and `value` columns. Session data is stored as JSON in the `value` column under keys like `sessionData.<sessionId>`.

The query pattern:
```sql
SELECT key, value FROM ItemTable WHERE key LIKE 'sessionData.%'
```

Parse the JSON value to extract message count and session timestamps.

**Important:** The exact schema may vary between Cursor versions. The implementation must:
1. Try the known schema first
2. If the query fails (SQL error, missing table, locked database), fall back to the mtime probe and log a warning to stderr
3. If the query returns rows but the JSON structure is unexpected, fall back to mtime and log a warning
4. Never crash the discipline report — all SQLite errors are caught and degraded gracefully

### File changes

- **Modify:** `src/core/discipline/transcripts/cursor.ts` — replace mtime probe with SQLite parsing
- **Modify:** `src/core/discipline/transcripts/types.ts` — extend `TranscriptRuntime` return type to include session/message counts (or use a new result type)
- **Modify:** `src/core/discipline/transcripts/index.ts` — handle new result shape
- **New:** `tests/core/discipline/transcripts/cursor-sqlite.test.ts`
- **Modify:** `CHANGELOG.md`

### Design decisions

- **`bun:sqlite` only.** Already in the stack. No new dependencies.
- **Fallback to mtime on any SQLite error.** If the schema changed, the database is locked, or permissions are wrong, we degrade gracefully to the old mtime behavior.
- **Isolated in one file.** No changes to other transcript resolvers (Claude Code, Codex).

## Feature 4: Obsidian Deep-Link in Explorer

### What it does

Double-click on a node in live-mode `o2b brain explorer` opens the corresponding `pref-X.md` file directly in Obsidian via `obsidian://open?path=<absolute_path>`.

### Implementation

1. **Server-side:** `buildLiveServer` passes `vaultPath` to the HTML template via a new placeholder `__VAULT_PATH__` (similar to `__GRAPH_JSON__`).
2. **Template:** Add `__VAULT_PATH__` placeholder and ~30 lines of JS:
   - On double-click of a canvas node or listbox item
   - Construct `obsidian://open?path=${encodeURIComponent(absolutePath)}`
   - Attempt to open via `window.open(uri, '_blank')`
   - Fallback: copy absolute path to clipboard + show tooltip "Obsidian not detected — path copied"

3. **Path construction:** `<vaultPath>/Brain/preferences/pref-<id>.md`

Only active in live-mode. Export-mode has no vault path, so no deep-link.

### File changes

- **Modify:** `src/core/brain/explorer.ts` — pass `vaultPath` to template in live server mode
- **Modify:** `templates/brain-explorer.html` — add `__VAULT_PATH__` placeholder + double-click handler
- **New:** `tests/core/brain/explorer-deeplink.test.ts`
- **Modify:** `CHANGELOG.md`

### Design decisions

- **`path=` not `vault=`.** `obsidian://open?path=<absolute>` works regardless of how the vault is registered in Obsidian. `vault=<name>` requires the vault name to match Obsidian's registered name, which may differ from the directory basename.
- **Silent fallback.** If `window.open` doesn't trigger Obsidian (not installed, blocked by browser), copy path to clipboard and show a brief tooltip. No modal dialogs.
- **No server-side code for the fallback.** Pure client-side JS. The server only provides the vault path.
